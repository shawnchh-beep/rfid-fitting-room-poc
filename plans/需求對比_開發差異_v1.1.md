# RFID 試衣間需求對比與開發差異 v1.1

## 對比範圍
- 需求來源：`規格書/RFID_Fitting_Room_Demo_需求文件_v1.1_更新版.md`
- 目前程式：`public/index.html`、`public/js/main.js`、`public/css/style.css`、`api/rfid-webhook.js`、`api/bulk-products.js`

## 總結判定
- **開發方向整體一致**：已朝「前台互動 + 後台資料流」推進，並完成拖拉視覺化主畫面與二級備援工具。
- **主要差異在第二階段 API 與資料模型完整度**：目前前端拖拉已可視覺操作，但尚未把每次拖拉完整寫成標準化業務事件。
- **可接受的命名差異存在**：多數為命名口徑不同，不影響持續開發，可維持現有名稱並建立對照。

## B. 名稱差異對照表

> 原則：僅名稱差異者可維持現況命名，不強制重命名；以對照表統一語意。

| 需求文件名稱 | 目前程式或資料名稱 | 類型 | 是否僅命名差異 | 建議處置 |
|---|---|---|---|---|
| `sales_floor` | `RACK` | 狀態值 | ✅ 是 | 維持 `RACK`，文件註記對照 |
| `in_fitting_room` | `FITTING_ROOM` | 狀態值 | ✅ 是 | 維持現名 |
| `left_fitting_room` | 回到 `RACK`（presence 失效） | 狀態語意 | 🟡 近似 | 補事件 type `exit_fitting_room` |
| `alert_overstay` | `abnormal` 布林 + 文案 | 告警狀態 | 🟡 近似 | 可增 `alert_status` 欄位 |
| `product_items` | `inventory_items` | 資料表 | ✅ 是 | 維持 `inventory_items`，文件加 alias |
| `epc` | `epc_data` | 欄位名 | ✅ 是 | 維持 `epc_data` |
| `event_time` | `timestamp` | 欄位名 | ✅ 是 | 維持 `timestamp` |
| `exited_at` | `left_at` | Session 欄位 | ✅ 是 | 維持 `left_at` |
| `sale_completed` | `SOLD` 或 checkout 流程 | 事件/狀態 | 🟡 近似 | 第二階段補 `sale_completed` 事件 |
| `event_source` | 尚未統一（模擬/reader） | 事件欄位 | ❌ 否 | 需新增欄位或 metadata |
| `product_name` | `name`/`name_en`/`display_name` | 顯示欄位 | ✅ 是 | 維持現名，UI 以 `display_name` 優先 |
| `entered_fitting_room_at` | `entered_at` | 欄位名 | ✅ 是 | 維持 `entered_at` |
| `last_seen_at` | `last_seen_at` | 欄位名 | ✅ 一致 | 無需處理 |
